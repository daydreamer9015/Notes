1. MSA (MicroService Architecture)란?
    1) 하나의 큰 어플리케이션을 여러 개의 작고 '독립적인 서비스'로 개발하고 배포하는 소프트웨어 개발 아키텍처를 의미함
    2) 모노리틱 아키텍처
        - 일반적으로 백엔드 서버를 구성할 때 활용하는 구조
        - 프로젝트 하나에 모든 API를 전부 구현하는 방식임
    3) 마이크로서비스 아키텍처 (MSA)
        - 서비스에 필요한 API들을 하나의 프로젝트가 아닌 여러 개의 프로젝트로 나눠서 구현하는 방식임
        - 결제 관련 API들끼리 모아놓은 프로젝트와 인증 관련 API들끼리 모아놓은 프로젝트, 상품 관련 API들끼리 모아놓은 프로젝트를 분리해서 구성함
        - 독립적인 분리된 하나의 프로젝트를 서비스(Service)라고 함

2. MSA 핵심 특징 (장점)
    1) 가장 핵심 특징은 독립성임
        - 독립적으로 분리해서 프로젝트를 구성함
    2) 서비스 별 독립적인 배포
        - 배포 사이클이 빨라짐
            - 서로 의사소통할 인원이 적기 때문에 프로젝트 진행이 빠름
            - 팀이 독립적으로 개발/배포를 진행할 수 있음
            - 코드의 양이 비교적 작기 때문에 코드 작성, 테스트, 빌드 과정의 시간이 훨씬 적게 걸림
    3) 데이터베이스의 독립적인 분리
        - 대규모 트래픽 처리 용이
    4) 서비스를 독립적으로 분리
        - 장애 전파를 최소화

3 언제 MSA를 도입하는 게 적절한가?
    1) MSA 도입 사례
        - 넷플릭스
            - 시스템의 안정성와 탄력성 확보를 위해 MSA로 전환함
            - 매우 빠른 배포 속도
            - 각각의 기능을 독립적으로 확장
            - 대규모 트래픽을 안정적으로 처리
        - 아마존
            - 유연하고 빠르게 확장
            - 다양한 신규 서비스를 신속하게 배포
        - 우버
            - 새로운 기능 추가하거나 수정할 때마다 복잡성이 증가하고 배포 시간이 길어짐
            - MSA로 전환
                - 개발 속도를 높이고, 각 서비스의 확장성을 확보
    2) MSA로 전환하는 이유
        - 신속한 배포
        - 대규모 트래픽을 안정적으로 처리
        - 시스템 안정성 확보
    3) MSA로 언제 전환하는 게 좋을까?
        - 프로젝트 규모가 크거나 또는 개발자 인원이 너무 많아서 배포 사이클이 느려진 경우
        - 모노리틱 아키텍처에서는 도저히 감당할 수 없는 트래픽이 발생하는 경우
        - 모노리틱 아키텍처로 인한 장애 전바의 횟수가 잦거나 한번의 장애 전파도 서비스에 치명적인 경우
